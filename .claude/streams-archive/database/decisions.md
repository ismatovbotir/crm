# Database Stream — Decisions Log

Append-only лог архитектурных решений по БД и моделям.

---

## 2026-04-27 — MySQL (а не PostgreSQL)
**Решение**: Используем MySQL 8.x.
**Причина**: проще хостинг в Узбекистане, привычно команде, есть в Laragon.
**Trade-off**: PostgreSQL имеет лучше поддержку JSON и concurrent transactions, но для B2B CRM не критично.

## 2026-04-28 — Spatie Permission для ролей
**Решение**: `spatie/laravel-permission` для ролей и прав.
**Причина**: де-факто стандарт в Laravel, гибкие permission через БД, хорошо тестируется.
**Альтернатива**: гардовая Laravel Gate — недостаточно гибко для B2B с динамическими ролями.

## 2026-04-28 — Soft deletes для критичных сущностей
**Решение**: `Customer`, `Lead`, `Quote`, `Invoice`, `Ticket` — soft deletes (`deleted_at` колонка).
**Причина**: бизнес-данные нельзя терять. История важна для аудита.
**Не используем soft deletes**: для справочников (sources, categories).

## 2026-04-28 — business_types как справочник (а не enum)
**Решение**: `business_types` — отдельная таблица (id, name, slug, is_active, sort_order). Customer и Lead ссылаются через FK.
**Причина**: лиду проекта нужна возможность редактировать типы бизнеса в admin-панели без миграций.
**Trade-off**: один JOIN при выводе списков. Минус 5% производительности — но допустимо.

## 2026-04-28 — Мультиязычность товаров (RU + UZ)
**Решение**: `product_translations` (product_id, locale, name, description). Products.name остаётся как RU-дефолт (fallback). Для UZ — translation row.
**Причина**: рынок Узбекистана требует UZ-каталог для портала.
**Trade-off**: дополнительная таблица, нужна логика подмены локали. Стандартный паттерн.

## 2026-04-28 — Specs по предопределённым полям категории
**Решение**: `category_attributes` (определение полей для категории: key, name, type, options, unit) + `product_attribute_values` (значения для конкретного товара). Убираем `products.specs` JSON.
**Причина**: фильтрация в каталоге по характеристикам, валидация ввода, единые названия атрибутов.
**Trade-off**: 2 дополнительные таблицы и больше JOIN'ов при выводе товара. Но даёт правильный функционал каталога.

## 2026-04-28 — Денежные поля как DECIMAL(15,2) UZS, отдельный USD
**Решение**: суммы хранятся в основной валюте (UZS) как `DECIMAL(15,2)`. Если запись в USD — отдельная колонка `currency` + `amount_usd`.
**Причина**: UZS даёт большие числа (миллионы), float теряет точность. Двух-валютный учёт обязателен для импорта.
